# Introduction

Evaliphy is the first QA-centric, end-to-end RAG (Retrieval-Augmented Generation) testing framework. It treats RAG systems as a black box, allowing engineers to validate AI behaviour through clear, readable assertions — without needing to understand retrieval pipelines, prompt engineering, or ML metrics.

## Motivation

Most teams building RAG features are not ML teams. They are product engineers and QA engineers who know how to ship software, write tests, and maintain CI pipelines.

But when it comes to evaluating whether their RAG system actually works, there is no obvious place to start. The questions are real:

- Is the response faithful to the retrieved context?
- Is it actually answering what the user asked?
- Did the last deployment quietly break something?

The existing tools are powerful but built for a different audience. For a QA engineer who just wants to know if the last deployment broke something:

- They assume familiarity with ML concepts
- They require Python environments
- They sit outside the normal engineering workflow

Evaliphy was built to close that gap. If you can write an end-to-end test for your API, you should be able to write an end-to-end test for your RAG system — using the same workflow, the same language, and the same CI pipeline.

## Why Evaliphy?

**It fits where your tests already live.**
- TypeScript-native — eval files sit in your repo alongside your other tests
- Runs in CI like any other test suite
- Produces reports your whole team can read

**You test your real API.**
- Makes HTTP calls to your actual running service
- Evaluates what comes back — not a notebook, not a dataset loaded in memory
- If your RAG system breaks in production, Evaliphy catches it the same way your E2E tests catch a broken UI

**The judges are built in.**
- Faithfulness, relevance, groundedness — the assertions that matter are shipped with the framework
- Call `expect(response).toBeFaithful()` and Evaliphy handles the rest
- No prompt writing, no LLM wiring, no configuration beyond pointing it at your API

**Configurable without being overwhelming.**
- Sensible defaults out of the box — works without touching the config.
- Override the judge model globally, per eval file, or per individual assertion(to be developed).
- Bring your own prompts if the built-in ones don't fit your domain.

**It speaks QA.**
- Assertion style will feel familiar if you have used Jest, Vitest, or Playwright
- No new mental model to learn
- You are applying what you already know to AI output instead of UI behaviour

## How it differs from tools like DeepEval and Ragas

DeepEval and Ragas are excellent tools built by people who understand RAG evaluation deeply. Evaliphy is not trying to replace them. The difference is the audience and the workflow.

| | DeepEval / Ragas | Evaliphy |
|---|---|---|
| Primary audience | ML engineers | QA engineers |
| Language | Python | TypeScript |
| How it works | Bring your data to the tool | Tool calls your running API |
| Workflow | Offline evaluation | CI pipeline |
| Setup | Dataset + notebook | Eval file + npm install |

If your team has ML engineers running evaluation experiments, DeepEval or Ragas are likely the right fit. If your team has QA engineers catching regressions before they ship, Evaliphy is built for that.

## After beta

The beta exists to validate one thing — does this workflow actually work for QA engineers in practice? Based on what we learn, the roadmap includes:

- Broader assertion library covering more RAG quality dimensions
- CI integrations for GitHub Actions, GitLab, and Jenkins out of the box
- Multi-turn conversation evaluation for chat-based RAG systems
- Custom assertion support with your own prompts and scoring logic
- Comparison runs to diff two versions of your RAG system against each other

Nothing on this list is promised. It reflects where we think Evaliphy should go based on the problems we set out to solve.
